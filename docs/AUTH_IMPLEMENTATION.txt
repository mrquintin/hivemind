================================================================================
WHAT CHANGED
================================================================================

Previously, both the admin and client apps used username-only login — no
password was required. Anyone who knew a cleared username could obtain a JWT
granting full API access, including triggering Anthropic API calls.

Now:
  - Both apps require username + password to log in.
  - Passwords are hashed with bcrypt (via passlib) and stored in the database.
  - Failed login attempts are rate-limited (5 per username per 5 minutes).
  - Error messages are generic ("Invalid credentials") to prevent username
    enumeration.
  - Default users are automatically seeded on first server startup.

================================================================================
SECURITY MEASURES
================================================================================

  1. Bcrypt password hashing (passlib CryptContext, auto-deprecated schemes)
  2. Login rate limiting: 5 failed attempts per username per 5-minute window
     returns HTTP 429. In-memory, thread-safe (threading.Lock).
  3. Generic error messages: "Invalid credentials" for both wrong username and
     wrong password — prevents username enumeration.
  4. Usernames normalized to lowercase on storage and comparison.
  5. is_active flag on User model for future account disabling.
  6. JWT format unchanged (sub, role, exp) — existing auth infrastructure
     (deps.py, role checks) continues to work without modification.
