Hivemind Product Description (v2 Direction)

Hivemind is a strategic decision system that combines rigorous theory with practical feasibility constraints so users can make high-stakes choices with both intellectual depth and real-world viability.

The product has three applications:
- Admin (Tauri + React): operator console for configuring agents, knowledge bases, simulation assets, and system settings.
- Cloud (FastAPI + PostgreSQL + Qdrant): central execution layer for retrieval, debate orchestration, practicality scoring, and audit logging.
- Client (Tauri + React): end-user interface for submitting problems and receiving recommendations with evidence and feasibility outcomes.

The architectural goal is to preserve analytical depth while reducing user-side complexity, runtime cost, and algorithmic brittleness.

---
Core Product Position

Traditional strategy education teaches frameworks but not reliable, repeatable application in real decisions. Hivemind operationalizes that process by:
1) retrieving relevant strategic knowledge,
2) generating and refining candidate recommendations,
3) stress-testing recommendations against practicality constraints,
4) returning transparent outcomes with audit metadata.

The system is intentionally designed to separate:
- theoretical insight generation,
- monitor-level synthesis,
- practicality feasibility gating,
so each function can be tuned independently.

---
Knowledge and Data Model

Before analysis, the cloud can access:
- operator-curated strategic knowledge bases,
- simulation assets,
- practicality/compliance knowledge,
- user-provided context text (when explicitly submitted).

Admin-managed uploads are purpose-specific:
- Theory frameworks: primarily .txt (with optional richer formats if enabled), embedded for RAG retrieval.
- Simulation programs: .py executable artifacts plus companion descriptive text for retrieval and tool selection.
- Practicality documents: primarily .txt constraints and scoring guidance, embedded for feasibility evaluation.

Document pipeline:
extract -> optional optimization -> chunk -> embed -> store metadata in PostgreSQL + vectors in Qdrant.

---
Client Experience (v2)

The default client experience is simplified and outcome-oriented.

Default controls (visible):
- Problem statement
- Sufficiency target (desired number of distinct conclusions)
- Feasibility strictness
- Analysis mode (Simple / Full)
- Effort level (Low / Medium / High)
- Optional decision profile selectors

Advanced controls (hidden by default):
- legacy algorithm knobs (density/similarity/revision) are treated as experimental and are not required for standard runs.

This design removes hyperparameter burden from users and shifts optimization responsibility to internal orchestration.

---
Execution Modes

1) Simple Mode (default baseline)
- single-pass theory generation,
- lightweight aggregation,
- practicality scoring,
- optional limited repair pass,
- lower latency and lower cost.

2) Full Mode (deep analysis)
- multi-round critique/revision,
- stronger monitor clustering pipeline,
- recommendation-level repair iterations,
- bounded by explicit runtime/token/call budgets.

Simple mode exists as a practical baseline for most requests; Full mode exists for high-importance analyses where deeper deliberation is worth additional cost.

---
Theory, Monitor, and Practicality Roles

Theory network:
- generates candidate strategic recommendations using RAG-grounded context and available simulations.

Monitor:
- synthesizes semantically overlapping recommendations,
- tracks distinct conclusion count against sufficiency,
- emits transparent clustering evidence.

Practicality network:
- scores each recommendation on feasibility,
- returns risks/challenges/mitigations,
- drives repair input for weak recommendations.

---
Veto and Repair Policy (v2)

Legacy behavior (full blind restart) is no longer the default.

New default behavior:
- practicality failures are handled recommendation-by-recommendation,
- each failed recommendation enters a bounded repair loop informed by practicality feedback,
- repaired recommendations are rescored,
- unresolved items are returned explicitly as failed_after_repairs rather than silently discarded.

Global restart is reserved as an exceptional fallback path, not the standard mechanism.

---
Convergence, Safety, and Cost Guarantees

Hivemind v2 requires deterministic stop conditions:
- max debate rounds,
- max repair iterations,
- max LLM calls,
- max total tokens,
- max wall-clock time,
- stagnation detection (no meaningful aggregation improvement across a configured window).

Every run must terminate with an explicit termination reason (for example: sufficiency_reached, completed_with_repairs, budget_exhausted, stagnation_early_stop).

This makes runtime behavior financially predictable and operationally safe.

---
Monitor Similarity (v2)

Similarity is treated as a first-class technical subsystem.

The monitor uses a layered approach:
1) fast embedding-based candidate grouping,
2) semantic adjudication for borderline cases,
3) canonical-form comparison for merge safety,
4) per-cluster confidence and rationale logging.

This reduces false merges/splits and makes aggregation decisions auditable.

---
Output Contract

Client receives:
- approved recommendations,
- vetoed/failed-after-repair recommendations,
- feasibility detail by practicality agent,
- retrieved evidence references,
- full audit trail,
- run statistics (tokens, duration, rounds, repairs, termination reason).

This supports both decision-making and governance/review.

---
Operational Direction

Hivemind is now explicitly designed to balance:
- recommendation quality,
- compute cost,
- termination reliability,
- user simplicity,
without requiring end-users to tune complex algorithmic internals.

In short: default simple behavior for consistency and affordability, optional full behavior for depth, and transparent controls around risk, cost, and convergence.
